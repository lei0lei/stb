# 项目整体逻辑和使用方式

这个仓库没有一个统一的“程序主流程”。它的整体逻辑是：每个核心文件都是一个可以单独复制进项目的库，测试和工具围绕这些库验证行为、生成辅助内容或维护发布资料。

## 使用逻辑

用户通常只需要复制某个 `stb_*.h` 文件到自己的项目。普通头文件包含只获得声明；在一个并且只能一个 C/C++ 源文件中定义实现宏后再包含该文件，才会生成函数定义。例如使用图像读取库时，常见写法是先定义 `STB_IMAGE_IMPLEMENTATION`，再 include `stb_image.h`。

这种模式解决的是分发和集成问题。传统库往往需要头文件、源文件、构建脚本和链接配置；stb 风格把这些压成一个文件，让用户可以直接放进已有工程。代价是单个文件很长，内部模块边界更多依赖注释、宏命名和私有函数前缀来表达。

## 编译逻辑

每个库都尽量减少外部依赖，主要依赖 C 标准库。平台差异通过预处理宏处理，例如 C/C++ 链接约定、编译器警告、SIMD 路径、内存分配函数替换和静态实现选项。部分库允许用户定义宏来替换 `malloc/free`、assert、I/O 回调或配置常量。

测试编译逻辑集中在 `tests/Makefile` 和若干测试源文件中。测试源文件会分别启用目标库实现宏，验证 C 编译、C++ 编译、图像格式、字体、sprintf、容器、噪声、体素等功能。fuzz 入口用于把随机或变异输入喂给解析类库，帮助发现崩溃和越界问题。

## 运行时逻辑

运行时层面，每个库都有自己的数据流。图像库从字节流到像素缓冲区；字体库从字体二进制表到字形轮廓再到位图；Vorbis 从 Ogg/Vorbis 比特流到 PCM；容器库从宏调用到动态内存块；文本编辑库从输入事件到编辑状态变化。

这些库之间一般没有强耦合。少数场景会组合使用，例如 `stb_truetype.h` 打包字形时可借助矩形装箱思想，图形测试可能同时使用图像加载和写出。项目整体强调“可组合但不强绑定”：用户按需拿走一个文件，不需要接受整个框架。

## 测试和维护逻辑

`tests/` 覆盖 API 行为、编译兼容性和格式回归。图像类测试会读取样本数据或构造输入，比较解码结果和预期行为；编译测试确保 C 和 C++ 模式都能通过；fuzzer 入口重点验证解析器面对异常输入时不崩溃。

`tools/` 的逻辑偏维护，例如生成 README 表格、检查尾随空白或处理字体/Unicode 辅助数据。它们不是最终库的一部分，但反映了这个仓库的发布方式：核心库保持单文件，围绕它们的文档、测试和发布信息可由工具辅助生成。

## 阅读顺序建议

如果想理解项目整体，先读 `README.md` 和 `docs/stb_howto.txt`，理解单文件库理念；再读一个小库如 `stb_easy_font.h` 或 `stb_perlin.h`，熟悉声明区和实现区；随后再进入大库如 `stb_image.h`、`stb_truetype.h`、`stb_image_resize2.h` 或 `stb_vorbis.c`。测试文件适合在读完 API 后回头看，因为它们展示了真实调用方式。
