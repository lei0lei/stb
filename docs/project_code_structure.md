# 项目整体代码结构解说

`stb` 是一组面向 C/C++ 的单文件库集合。它不是一个传统意义上有统一运行时入口的应用，而是把多个彼此独立的小型库放在同一个仓库里，每个库都尽量做到“复制一个文件即可使用”。因此，理解这个项目时应先把它看成库集合，再看每个单文件库内部的声明区、实现区、测试区和工具区。

## 顶层结构

根目录下的 `stb_*.h` 和少量 `*.c` 是项目核心。大多数文件采用单头文件模式：默认包含时只暴露类型、宏和函数声明；只有在某个 C/C++ 编译单元中先定义对应的 `*_IMPLEMENTATION` 宏，再包含该头文件，才会编译实现代码。`stb_vorbis.c` 例外，它本身是一个 C 源文件，但也保持类似“单文件交付”的理念。

`tests/` 目录放测试、示例和 fuzz 入口。这里的文件通常会定义某个库的实现宏，然后 include 对应单文件库，再构造输入检查 API 行为。`tests/Makefile` 展示了项目最直接的编译方式：用普通 C/C++ 编译器把多个测试文件和目标库一起编译，不依赖复杂构建系统。

`tools/` 目录放维护脚本和生成工具，例如生成 README、检查尾随空白、处理 Unicode 数据或构建矩阵。这些工具服务于仓库维护，不是最终用户集成 stb 库时必须使用的代码。

`deprecated/` 保存旧版本或已不推荐的新项目继续使用的代码。它们仍可作为历史参考或兼容用途，但主要关注点应放在根目录当前版本的库。

`stb_image_resize_test/` 是图像缩放库的专项测试和对比代码，用来验证新旧 resize 实现的行为、性能和结果质量。

## 单文件库的内部组织

每个核心库通常按以下层次组织：

1. 顶部注释说明库名、版本、用途、作者、许可证和快速用法。
2. 公共 API 区声明用户会直接调用的函数、类型和配置宏。
3. 条件编译区根据平台、编译器、用户宏和 C/C++ 差异选择不同代码路径。
4. 实现区在 `*_IMPLEMENTATION` 宏启用时编译，包含内部类型、私有函数、算法实现和错误处理。
5. 许可证文本通常放在文件末尾，保持单文件分发时法律信息完整。

这种组织方式牺牲了一些模块拆分的清晰度，但换来极低集成成本：用户不需要安装库、不需要链接多个目标文件，也不需要维护额外构建脚本。

## 主要库分组

图像相关库包括 `stb_image.h`、`stb_image_write.h`、`stb_image_resize2.h`、`stb_dxt.h` 和旧版 `deprecated/stb_image_resize.h`。它们覆盖读取、多格式解码、图像写出、缩放重采样和纹理压缩。

字体和文本相关库包括 `stb_truetype.h`、`stb_easy_font.h`、`stb_textedit.h`。它们分别处理 TrueType 字体解析与光栅化、简单调试文字几何生成，以及文本编辑控件内部状态机。

数据结构和工具类库包括 `stb_ds.h`、`stb_sprintf.h`、`stb_c_lexer.h`、`stb_include.h`、`stb_leakcheck.h`。这些文件提供动态数组/哈希表、格式化输出、词法分析、递归 include 展开和轻量内存泄漏检查。

图形和游戏开发相关库包括 `stb_rect_pack.h`、`stb_perlin.h`、`stb_voxel_render.h`、`stb_tilemap_editor.h`、`stb_herringbone_wang_tile.h`、`stb_connected_components.h`。它们处理矩形装箱、噪声、体素渲染、瓦片地图编辑、Wang tile 生成和网格连通性维护。
